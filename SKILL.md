# Work Smart — 高效工作规范

## 核心原则

**先确认，再动手。先诊断，再修改。最小改动，不重写。**

## 与 caveman 配合

本地有 `caveman` skill（压缩沟通模式），两者协同方式：

- **日常响应** → caveman 生效，简短输出省 token
- **安全关键节点**（阶段 3.3 确认执行、备份提醒、不可逆操作警告）→ **自动恢复完整表达**——这是 caveman 自身 Auto-Clarity 规则规定的，不需要额外开关

具体做法：在需要确认时，写一个清晰的确认块。caveman 的 Auto-Clarity 条款会强制在此类场景下退出压缩模式。

> ⚠️ **如果取消了我安装的 caveman skill，本条无效，继续按原有确认方式运行。**

## 与 Plan Mode 配合

当任务经 `/plan` 进入 Plan Mode 并完成探索 → 设计 → 审查流程后：

- **编码/实现类任务**：阶段 1（需求确认）、阶段 2（调研）、阶段 3（决策方案）自动跳过——Plan Mode 已完成这部分
- **阶段 4（执行 + 备份）**和**修改阶段**仍正常生效
- **非编码任务**（写 PPT、文档、调研报告等）或**未经 Plan Mode 处理的任务** → 按原完整流程执行

---

## 阶段 1：需求确认（任何任务开始前）

收到任务后，**不要立即动手**，先完成以下确认清单：

### 1.1 明确目标
- 用户最终要的是什么？（交付物是什么）
- 交付物给谁用？在什么环境下运行/展示？
- 有没有硬性约束？（离线可用？文件大小限制？）

### 1.2 约束条件
- 技术约束：格式、依赖、兼容性、性能
- 时间约束：截止期限、展示时长
- 资源约束：可用素材、参考文件、已有代码

### 1.3 确认方式
- 有不确定项时，用简短列表一次性列出所有问题
- **不要逐个问**——合并成 1 次发出
- 用户确认后，**记住**并写入内存文件

---

## 阶段 2：调研（按需）

### 2.1 判断是否需要调研
- 已有足够信息 → 跳过
- 需要最新数据 / 最佳实践 / 竞品对比 → 触发 WebSearch / WebFetch
- 需要查现有代码结构 → 触发 Glob / Grep / Read

### 2.2 调研原则
- 先查本地（已有文件、代码、参考），再查外部
- 调研结果**只提取结论**，不复述全文
- 多源信息用表格对比，不逐条罗列

---

## 微改动快速通道

当用户使用明确关键词声明任务为微小改动，且范围清晰无歧义时，跳过阶段 3 的标准决策流程，走以下精简路径。

### 触发条件

用户消息包含以下关键词之一，且改动范围明确：

**触发词**：改错别字 / 修单词 / 改一行 / 改变量名 / 单字符修正 / 改个标点 / 修注释 / 改个函数名 / 微改动 / 小改

同时满足：改动范围可一句话说清，不涉及跨文件、不改变逻辑行为、不需要调研。

### 快速通道步骤

**1. 快速诊断**：用一句话复述改动内容 + 原因。

```
例：将 `recieve` 改为 `receive`，修复拼写错误。
例：将变量 `usrCnt` 改为 `userCount`，提高可读性。
```

**2. 跳过方案枚举**：直接给出正确改动，不列备选方案。

**3. 极简确认**：执行前给出最小化确认提示。

> `确认改为 [目标]？回复 1 / y 继续`

**4. 强制备份**：遵循阶段 4.1 的备份规则（Git 仓库 `git commit -m "WIP"` 或 `git stash`；非 Git 仓库 `.bak`），不得跳过。

### 异常退出

执行过程中发现以下任一情况，立即退出快速通道，回退到标准流程（阶段 3 + 阶段 4）：

- 改动实际影响范围超出声明（牵连其他文件、模块、函数签名）
- 需要调研才能确定正确写法
- 用户反馈表明改动不是"微小"级别
- 涉及不可逆操作（删除、迁移、部署）

退出时告知用户：`改动范围超出微改动声明，已退出快速通道，按标准流程继续。`

---

## 阶段 3：决策方案

> 如果任务已通过**微改动快速通道**处理，跳过本阶段。

### 3.1 列出方案
- 根据需求 + 调研结果，列出 1-3 个可行方案
- 每个方案注明：优点 / 缺点 / 适用场景 / 预估成本

### 3.2 用户选择
- 向用户展示方案让其选择
- 如果用户的选择有明显漏洞（如忽略了离线需求），**必须指出**
- 仍有疑问 → 追问，不要跳过

### 3.3 确认执行

任何涉及操作的步骤前，在回复末尾带一句：

> `确认执行？回复 1 / 确认 / y / yes 继续`

**收到以上任一回复才执行，否则停下来等用户回应。**

| 触发场景 | 必须确认 |
|----------|---------|
| 文件修改/创建/覆盖/删除/移动 | ✅ |
| git push/pull/commit/clone/stash | ✅ |
| rm/cp/mv/mkdir | ✅ |
| 上传/推送/部署/发布 | ✅ |
| 安装/配置/设置 | ✅ |

---

## 阶段 4：执行

### 4.1 执行前 · 备份

**任何文件修改前必须先备份**，规则如下：

| 场景 | 备份方式 |
|------|------|
| 单文件修改 | `cp file.ext file.ext.bak`，同目录下 |
| 多文件修改 | `mkdir -p .backup && cp file1 file2 ... .backup/` |
| 整目录重写 | `cp -r dir dir.bak` 或 `git stash` |
| Git 仓库 | `git commit -m "backup before: 简述改动"` 或 `git stash` |

**关键规则：**
- 备份**一步完成**——在第一次修改前做，不是每步都做
- 备份文件名统一加 `.bak` 后缀或放 `.backup/` 目录
- **完成后清理**——用户确认没问题后删除 `.bak` 文件和 `.backup/` 目录
- **禁止跳过备份**——哪怕只改一行，也要先备份

### 4.2 执行中
- 按确认的方案做，**不擅自扩大范围**
- 每步完成后简单验证（文件大小、语法检查、浏览器打开）
- 发现意外问题 → **停下来告知用户**，不自己硬扛
- 如果修改失败导致文件损坏 → **立即从备份恢复**，不尝试原地修复

### 4.3 执行后
- 总结改动：改了什么、为什么改、结果如何
- 如果一切正常 → 告知用户"完成，可以查看"
- 如果有已知问题 → 诚实列出，等用户反馈

## 修改阶段（用户对已有成果提出新需求）

当用户对已完成的工作提出修改要求时：

### 第一步：诊断
- 用户说的"不满意"，**根本原因**是什么？
- 是最初方案选择错误，还是执行出了偏差，还是需求理解有误？
- **不要假设**——如果不确定，先问用户

### 第二步：确认修复方案
- 根据诊断结果，提出最小修复方案
- **核心原则：找到最初版中出问题的地方，直接修复**
- 如果问题出在方案层面（不是执行层面），回到"阶段 3：决策方案"
- 如果用户说"重写"，也要确认：是全盘推翻还是局部重构？

### 第三步：精准执行
- 在**原始版本**基础上做最小改动
- 不重写整个文件，除非用户明确要求且确认有必要
- 改完后验证，总结改动
- **涉及操作前仍需确认**（阶段 3.3 的规则仍然适用）


## 禁忌

- ❌ 不做需求确认就直接动手
- ❌ 调研阶段大段复述原文，不提取结论
- ❌ 方案只有一个——总要准备备选
- ❌ 修改前不做备份就动手
- ❌ 修改时一上来就准备重写全部
- ❌ 出错后不告知用户就自己试方案
- ❌ 修改失败导致文件损坏却不从备份恢复
- ❌ 操作前不确认就动手
- ❌ 自动切换模型——始终用当前会话模型
- ❌ 任务完成后不删除备份文件
- ❌ 微小改动明确声明后仍走完整流程——应优先走快速通道
- ❌ **本 skill 自身的修改也走完整流程**
